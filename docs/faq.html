What is the CIP4 JDF Library?

The CIP4 JDF Library is an SDK that provides support for CIP4s Job Definition Format JDF. The SDK is based on the Apache Xerces XML parser (see xml.apache.org) and contains high level JDF and JMF manipulation classes as well as typesafe access to JDF attributes and elements. The initial implementation is designed to be platform independent. It is written in C++ in a Microsoft Visual C++ 6.0 environment and updated for Microsoft Visual C++ 7.1. It comes with HTML documentation and a host of sample applications, including a complete JDF and JMF validator.

What Functionality does the CIP4 Library provide?

The library provides various degrees of support:

File and Network abstraction layer for windows.

Typesafe access to various elements of JDF:

JDF Nodes
JDF Resources
JMF Messages

High level support for JDF mechanisms:

Spawning + Merging
Link references
Partitioned reources
Validation

Who may use the CIP4 Library and for what Purposes?

Usage of the SDK is free to all for purposes of research, in-house use and testing. Commercial usage (selling SDK based products for profit) is limited to CIP4 full and partner members. For details, please contact info@cip4.org.

Where do I get support?

Developer support is available via an open eMail forum, JDFDev@cip4.org. CIP4 members may also participate in CIP4s Tools + Infrastructure working group. A set of simple sample applications can be found in the /tests folder of the source code.

Where is the 30.000 foot design description?

A short usage description can be found at JDF Library Classoverview . Note that this is work in progress.

What other packages are required for using the CIP4 Library ?

Xerces XML Parser

The Xerces C++ XML parser is also required. It should be version 2.0 or higher. Prior versions use a completely different interface.

OpenSSL

The sources can be found at http://www.openssl.org/source/openssl-0.9.6.tar.gz.

LIBPNG +ZLIB

Graphics support can be found at http://www.libpng.org/pub/png.

STL

An open source version of the Standard Template Library - STL support can be found at http://www.stlport.org. Include the stlport directory in the external directory to enable STLPort. No further modifications in the VC6 dsps are necessary.

Which Operating System platforms are supported?

Microsoft 32 bit windows using MSVC 6.

DSP files are provided. The dsp files are provided for static libraries, using the MSVC STL and the STLPort STL. The MSVC version has been more thoroughly tested. A DLL version is provided for the the STLPort STL. No DLL version using the MS STL is provided due to the incompatibility of MS STL and DLLs (see MS Knowledge Base q172396)

Microsoft 32 bit windows using MSVC .NET 7.1

Vcproj files are provided. The vcproj files are provided for static libraries. No STLPort STL is needed.

Linux (Suse 8.0)

A Makefile is provided and compiles with gcc 2.9x and 3.x. The Linux Port is still experimental. The test files compile and link and should work correctly.

Macintosh OS-X

A set of project files are provided and compile with gcc 2.9x and 3.x. The Mac Port is still extremely experimental. Note that due to a known bug in the STLPort library, the mac Port does not link correctly.

What does CheckJDF Do?

CheckJDF is not a completed bug-free product

CheckJDF is a demonstration tool / proof of concept for the CIP4 JDF Library. It validates a JDF using the TypeSafe validation classes in the CIP4 JDF Library.

CheckJDF does some additional validation compared to the pure Schema validation.

What are the changes w.r.t the 1.0 release?

JDF 1.1 schema support

This version is based on a pre-release of the JDF 1.1 schema.

Clean-Up of GetElement

The low level routine GetElement(...,bool bCreate,...) was split into two routines:
GetElement()const;
GetCreateElement()
This removes some undesired parameter overwriting features of the old version.

Clean-Up in Messaging

the routines JDFJMF(A|C|Q|R|S)XXX were removed and the functionality was included in JFDMessage. The interfaces remain intact when
JDFJMFAXXX() -> JDFAcknowledge()
JDFJMFCXXX() -> JDFCommand()
JDFJMFQXXX() -> JDFQuery()
JDFJMFRXXX() -> JDFResponse()
JDFJMFSXXX() -> JDFSignal()
The performance is slightly degraded but ~200 classes are replaced by 5, which seams a reasonable trade-off.

Clean-Up in Node Classes

the routines JDFNodeXXX were removed and the functionality was included in JDFNode. The interfaces have been reduced during the change. The change must be implemented as follows. class JDFNodeXXX --> class JDFNode.

JDFNodeXXX.GetYYY()-> JDFNode.GetMatchingResource(JDFElement::elm_YYY)
JDFNodeXXX.GetYYYLink()-> JDFNode.GetMatchingLink(JDFElement::elm_YYY)
JDFNodeXXX.AppendYYY()-> JDFNode.AppendMatchingResource(JDFElement::elm_YYY)
JDFNodeXXX.LinkYYY()-> JDFNode.LinkMatchingResource(JDFYYY)

Clean-Up in Generated Typesafe Classes

Non-TypeSafe routines have been removedfrom the generated typesafe classes:

NumXXX()
HasXXX()
RemoveXXX()
GetXXXVector()
Replace these with the appropriate generic routines and access the elements using JDFElement::elm_XXX and JDFElement::atr_XXX strings.

Changes for Xerces 2.x

Inheritence model

XMLDoc has been split into XMLDoc and JDFParser
KNode has been completely removed. Only KElement exists.
KNodeList has been completely removed. Use vElement instead.
KElement contains an opaque pointer to DOMElement and no longer exports DOM_Element functionality.

STL

The STL has been removed from the interfaces. STL is still used to implement functionality (vectors, maps, strings). The implementation is hidden through an opaque pointer in the interfaces. Thus only a subset is accessible.
